Touchpoints

Touchpoints are processes that are meant to both construct and deconstruct software. Destructing involves attacking, exploiting, or breaking your software in order to find vulnerabilities, while constructing involves building functionalities and designing defense mechanisms to protect software from vulnerabilities. There are seven touchpoints that we will be examining: abuse case, security requirements, risk analysis, risk-based security testing, code review, penetration testing, and security operations.

Abuse Cases

  • Many projects create use cases to better understand how a system is supposed to work. Abuse instances help anticipate what a system will do when it is attacked. It is extremely effective to consider misuse before it occurs
  • Taking an abstract view of your system architecture and brainstorming any possible methods your software could be attacked is another approach to look at abuse cases.

Security Requirements

  • How secure does your system have to be? Who is going to attack it, and why would they do so? Above all, how much can you afford to do about it?
  • Think of it in terms of having a door with a lock for a home. While one would want the door and lock to be as secure as possible most would not spend the money to make the door bombproof. During this process you need to understand your security needs and wants.

Risk Analysis

  • The goal of risk analysis is to detect, rank, and minimize design defects. In most software, design faults account for 50% of software security problems.
  • This is considered to be one of the two most important touchpoints

Risk-based Security Tests

  • Testing security functionality, probing design flaws identified in risk analysis, and regression testing against security bugs make up this process.

Code Review

  • This process helps find any bugs with the software that could be taken advantage of by an attacker. One example of a bug error is not allocating a proper amount of space to a buffer and causing it to overflow.
  • Tools like Fortify are a great way to automate reviewing source code.
  • Bugs account for the other 50% of secure software defects, which makes this process the second most important touchpoint.

Penetration Testing

  • Progressively investigating a running system for security flaws is a crucial activity, but it frequently occurs after the fact, making solutions much more complex and expensive.
  • Pen testing is important for designing secure software but should not be the only touchpoint being used.

Security Operations

  • Even a system that is designed for security, properly pen tested and debugged during development will be subjected to attack.
  • We are all human, and it is sometimes impossible to account for every possible vulnerability. So, this strategy is to implement is infosec or dev team to keep an eye on a running system.
